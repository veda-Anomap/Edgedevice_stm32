## [v1.2.1] - 2026-03-20
### 외부/내부 워치독 최종 튜닝 + 큐 복구 반영

변경 전 상태/문제
- 워치독 로직은 동작했지만, IWDG 타임아웃(약 5초) 대비 모니터 miss 임계가 보수적으로 작아
  스케줄링 지터 상황에서 불필요한 리셋 가능성이 있었음.
- `.ioc` 재생성 과정에서 `control_queue` 길이가 `64 -> 16`으로 내려가 버려 burst 구간 내성이 약해질 수 있었음.
- 문서에 최신 워치독 파라미터(10초/3초/60초, miss 임계)가 명시되지 않았음.

왜 바꿨는지
- 외부 릴레이 복구 로직과 내부 IWDG 리셋 정책을 실제 운용 지터까지 고려해 안정적으로 맞추기 위함.
- RPi 명령 burst 상황에서 명령 유실/ACK fail 가능성을 낮추기 위함.

무엇을 어떻게 바꿨는지
- `SYS_MONITOR_MISS_MAX`를 `4`로 조정(1초 주기 기준, 4회 연속 미갱신 시 refresh 중단).
- `StartDefaultTask`의 grace/relay 대기 구간을 공통 helper 함수로 정리:
  - `rpi_wdg_wait_seconds_and_mark_alive()`
  - `rpi_wdg_sync_heartbeat_baseline()`
- `control_queue`를 코드/IOC 모두 `64`로 복구.
- README에 watchdog 정책과 큐 길이 정보를 최신 코드 기준으로 업데이트.

변경 후 기능/안정성 개선 효과
- 워치독 오탐 리셋 여유가 증가해 장시간 운용 안정성이 개선됨.
- 릴레이 컷/복구/재-grace 흐름의 코드 가독성과 유지보수성이 향상됨.
- burst 명령 수용성이 이전 수준으로 복구됨.

영향 파일
- `Core/Src/main.c`
- `irrled.ioc`
- `README.md`

## [v1.1.4] - 2026-03-19
### 수동 모터 명령 버스트 완화 + 서보 안전 범위 축소

변경 전 상태/문제
- 수동 명령(`w/a/s/d`)이 짧은 시간에 몰리면 queue의 모든 명령이 연속 적용되어 서보가 과격하게 움직일 수 있었음.
- 기계적 종단 근처에서 급격한 왕복/충돌 느낌이 발생할 수 있었음.

왜 바꿨는지
- 실제 하드웨어에서 “획 도는 현상”과 극단 위치 떨림을 줄이기 위해, 제어를 부드럽게 제한할 필요가 있었음.

무엇을 어떻게 바꿨는지
- 수동 이동 명령을 coalescing 처리(최신 1개만 유지)하도록 `app.c`에 반영.
- `MANUAL_CMD_MIN_INTERVAL_MS`를 두어 최소 적용 간격을 강제.
- PAN/TILT PWM 한계를 기존보다 안쪽으로 조정(`motor_ctrl.h`)해 기계적 여유를 확보.

변경 후 기능/안정성 개선 효과
- 입력 버스트 시 서보가 한 번에 몰아 움직이는 현상이 완화됨.
- 종단 충돌 가능성 감소, 체감 안정성 증가.

영향 파일
- `Core/Src/app.c`
- `Core/Inc/motor_ctrl.h`

## [v1.1.3] - 2026-03-18
### UART1 파서 강건화(프레임 경계/타임아웃/큐)

변경 전 상태/문제
- UART1으로 데이터가 연속 유입될 때 프레임 경계가 흐트러지면, 잘못된 CMD/LEN을 기반으로 파서가 오동작할 수 있었음.
- 프레임 조립 도중 바이트 유입이 끊기면 상태머신이 중간 상태에 머물러 다음 정상 프레임까지 연쇄 실패가 발생할 수 있었음.
- 단일 완료 버퍼 구조에서는 burst 구간에서 뒤늦게 들어온 정상 프레임이 유실될 여지가 있었음.

왜 바꿨는지
- 라즈베리파이 연동 시 간헐적으로 발생하던 응답 실패를 줄이고, 프레임 손상 후 자동 복구 시간을 짧게 만들기 위해서.

무엇을 어떻게 바꿨는지
- CMD allow-list(`0x04`, `0x05`)를 적용해 유효하지 않은 타입은 즉시 폐기하도록 변경.
- `PROTO_RX_TIMEOUT_MS` 기준 타임아웃 리셋을 넣어 조립 중단 프레임이 다음 프레임에 영향 주지 않도록 분리.
- 완료 프레임 큐를 다중 슬롯 구조로 바꿔 burst 입력에서도 처리 여유를 확보.
- 바이트 조립 단계와 프레임 처리 단계를 분리해 디버깅과 장애 원인 추적이 가능하도록 정리.

변경 후 기능/안정성 개선 효과
- 입력 품질이 일시적으로 나빠져도 파서가 빠르게 정상 상태로 복귀.
- burst 구간 프레임 유실 가능성 감소.
- 통신 실패 원인(입력 손상/포맷 오류/처리 지연) 분리가 쉬워짐.

영향 파일
- `Core/Src/app.c`
- `Core/Src/main.c`

변경 후 기능/안정성 개선 효과
- “라인 품질 문제”와 “프레임 포맷 문제”를 빠르게 분리 가능.

## [v1.1.1] - 2026-03-18
### UART1 에러 복구 루틴 추가

변경 전 상태/문제
- FE/NE/PE/OE 등의 UART 오류가 한 번 발생하면 RX 인터럽트 재암이 실패하는 경우가 있었고,
  이 상태가 누적되면 UART1 수신이 장시간 멈출 수 있었음.

왜 바꿨는지
- 통신 라인 노이즈나 순간 타이밍 오류가 시스템 전체 통신 중단으로 번지지 않도록 복구 경로를 강제하기 위해.

무엇을 어떻게 바꿨는지
- 오류 발생 시 플래그 클리어 -> `HAL_UART_Receive_IT` 재암 순서로 기본 복구.
- 재암 실패 시 abort 후 재시도 루틴을 추가해 하드락 상태를 피하도록 처리.
- 복구 횟수 카운터를 남겨 런타임 상태에서 복구 성공/실패 추세를 추적 가능하게 구성.

변경 후 기능/안정성 개선 효과
- 일시적 UART 오류 후 자동 복구 확률이 크게 증가.
- 통신 중단 상태가 길게 지속될 위험 감소.

영향 파일
- `Core/Src/app.c`

## [v1.0.0] - 2026-03-18
### 마이크 경로 ADC -> I2S 전환 및 신호 안정화 반영

변경 전 상태/문제
- ADC 기반 설계와 I2S 전환 코드가 혼재되어 기준점/스케일/노이즈 처리가 불명확했음.

왜 바꿨는지
- INMP441(I2S) 기반으로 방향 감지 성능을 유지하면서 기존 제어 로직과 호환해야 했음.

무엇을 어떻게 바꿨는지
- I2S 데이터 언팩(24-bit signed) 후 기존 처리 경로로 연결.
- 동적 DC 오프셋 추적(DC blocking), 노이즈 게이트 적용.
- short/long window 기반 신호/노이즈 분리 및 SNR/비율 게이트 유지.
- ADC 경로는 조건부 컴파일 fallback으로 유지.

변경 후 기능/안정성 개선 효과
- I2S 환경에서 기준점 흔들림과 저레벨 노이즈 오탐이 감소.
- 기존 모터 제어 로직을 크게 깨지 않고 이식 가능해짐.

영향 파일

- `Core/Inc/mic.h`

- 제어/센서/수신 처리가 각각 태스크에서 수행된다는 점을 문서/코드에 명시.

변경 후 기능/안정성 개선 효과

## [v0.9.3] - 2026-03-05
### UART ISR 최소화 + RxTask 활성화

변경 전 상태/문제
- UART ISR 안에서 파싱 상태머신까지 처리해 인터럽트 체류 시간이 길었고,
  입력 burst에서 다른 태스크 지연 및 프레임 드롭 가능성이 있었음.

왜 바꿨는지
- ISR은 최소 동작만 수행하고, 파싱/비즈니스 로직은 태스크 컨텍스트에서 처리하는 RTOS 정석 구조로 바꾸기 위해.

무엇을 어떻게 바꿨는지
- ISR에서는 수신 바이트 1개를 `uart_rx_queue`에 넣고 즉시 리턴.
- `UartRxTask`에서 바이트 조립, 헤더/LEN/PAYLOAD 처리, 명령 분기 수행.

변경 후 기능/안정성 개선 효과
- 인터럽트 점유 시간 감소.
- burst 입력 상황에서도 전체 스케줄링 안정성 향상.
- 파싱 로직 테스트/수정 난이도 하락.

영향 파일
- `Core/Src/app.c`
- `Core/Src/main.c`

- RPi 프로토콜(JSON: `auto/unauto`, `w/a/s/d`)과 로컬 테스트 입력을 분리해 명령 충돌을 방지하기 위해.

무엇을 어떻게 바꿨는지
- 명령 allow-list를 명확히 정의하고 모드 전환 규칙을 정리.
- 자동/수동 모드에서 허용되는 입력을 분리해 예상치 못한 상태 전이 제거.

변경 후 기능/안정성 개선 효과
- 동일 입력에 대한 동작이 일관화됨.
- 모드 전환 직후 오동작 가능성 감소.

영향 파일

## [v0.9.1] - 2026-03-03
### AHT10 상태머신 + 이동평균(N=10)

변경 전 상태/문제
- 센서 값을 너무 자주 읽거나 변환 대기 없이 읽으면 self-heating과 오측정이 발생할 수 있었음.
- 단발 스파이크 값이 그대로 상위 상태값에 반영되어, 연속 로그에서 값이 튀는 문제가 있었음.

왜 바꿨는지
- 실제 운용 환경에서 안정적인 온습도 값을 확보하려면 측정 주기 제한, 변환 완료 확인, 필터링이 함께 필요했음.

무엇을 어떻게 바꿨는지
- 측정 주기를 최소 1초 이상으로 강제하고 기본 주기를 2초로 설정.
- 측정 명령 전송 후 변환 대기(약 80ms)를 두고 BUSY 비트를 재확인하도록 상태머신 구성.
- 최근 N=10 샘플을 FIFO 이동평균으로 처리해 순간 튐 값을 완화.

변경 후 기능/안정성 개선 효과
- 과측정으로 인한 센서 자체 발열 영향 감소.
- 스파이크 완화로 시계열 안정성 향상.
- 상위 제어/표시 로직에서 값 활용 신뢰도 증가.

영향 파일
- `Core/Src/aht10.c`
- `Core/Inc/aht10.h`

## 참고: 직접 테스트/분석 기반 개선 방향

- 인터럽트 최소화: ISR은 큐잉만, 파싱/응답은 태스크에서 처리.
- 태스크 분리: 제어 루프/센서 루프 분리로 지연 전파 차단.
- 프로토콜 안정화: CMD/LEN 검증, 프레임 타임아웃, 버스트 흡수.
- 진단 강화: UART 오류 카운터 분리로 원인 분석 속도 개선.
- 모터 안정화: burst coalescing + 최소 적용 간격 + 안전 범위 적용.
- 신호 안정화: I2S DC 블로킹/노이즈 게이트 + 윈도우 기반 SNR 게이트 유지.

# UART 안정화 에러 수정 보고서

문서 목적:
- 최근 발생한 "센서 요청 fail/성공 반복", "응답 몰림", "간헐 수신 불안정" 이슈를
  UART 경로 기준으로 분리해서 정리한다.
- 이번 문서는 기능 추가가 아니라 **에러 대응 수정 사항만** 기록한다.

적용 버전:
- v1.1.3 (2026-03-18)

대상 파일:
- `Core/Src/app.c`
- `Core/Src/main.c`

---

## 1. 장애 증상(수정 전)

- RPi에서 상태 요청(0x05)을 보낼 때 응답이
  - 어떤 구간에서는 실패처럼 보이고
  - 어떤 구간에서는 한꺼번에 몰려서 들어오는 현상 발생
- 모니터 로그에서 다음 패턴 확인
  - `Q:0`, `DR:0`, `O/F/N/P:0`인데 `IV/OV`가 증가
  - 물리 UART 에러보다 프레임 경계/길이 파싱 이슈 가능성이 높은 패턴
- 요청 버스트 구간에서 ACK fail 체감 발생 가능

---

## 2. 원인 분석 요약

1) CMD 동기 취약
- `WAIT_CMD` 상태에서 임의 바이트를 CMD로 수용
- 쓰레기 바이트가 끼면 파서 동기 붕괴 가능

2) 프레임 수신 타임아웃 부재
- 헤더/바디 수신 중 끊기면 상태머신이 꼬인 채 유지될 수 있음

3) 프레임 보관 슬롯 1개
- 완료 프레임 버퍼가 1칸이라 버스트 입력에서 유효 프레임 유실 가능

4) 처리 경로 결합
- UartRxTask에서 수신/파싱/응답까지 수행
- 수신 집중 시 제어 루프 반영 지연 가능

5) 상태 응답 레이트 제한 없음
- 짧은 시간에 0x05 요청이 반복되면 응답 트래픽이 급증
- 체감상 "갑자기 엄청 보냄" 현상 유발

6) 제어 큐 용량 여유 부족
- burst 시 `control_queue` 포화로 ACK fail 가능성 증가

---

## 3. 수정 내용 상세

### A. CMD 화이트리스트 적용

수정:
- `WAIT_CMD`에서 `0x04`, `0x05`만 유효 CMD로 수용
- 그 외 바이트는 즉시 폐기하고 invalid 카운트 증가

효과:
- 프레임 동기 복구가 빨라지고 연쇄 파싱 오류 완화

---

### B. 프레임 타임아웃 추가

수정:
- `PROTO_RX_TIMEOUT_MS = 100ms` 도입
- 프레임 조립 중 마지막 바이트 이후 100ms 초과 시 강제 reset

효과:
- 중간에 끊긴 프레임으로 상태머신이 고착되는 문제 방지

---

### C. 완료 프레임 저장 구조를 링큐로 변경

수정:
- 단일 완료 프레임 슬롯 -> `PROTO_FRAME_QUEUE_LEN = 8` 링큐
- head/tail/count 기반 push/pop 처리

효과:
- 요청 버스트 구간에서 단기 흡수 가능
- 유효 프레임 유실 확률 감소

---

### D. UART 처리 경로 분리

수정:
- `app_on_uart1_byte()`는 바이트 축적만 수행
- 실제 프레임 처리(`0x04/0x05`, ACK, 상태응답)는 `app_control_loop()`에서 실행

효과:
- 수신 경로가 가벼워짐
- 제어/응답 처리 시점이 일정해져 지연 편차 감소

---

### E. 상태 응답 레이트 제한 + 병합 응답

수정:
- `STATUS_REPLY_MIN_INTERVAL_MS = 100ms` 도입
- 간격 내 중복 0x05 요청은 `pending`으로 병합 후 다음 주기에 1회 응답

효과:
- 응답 폭주 완화
- 요청이 몰려도 트래픽이 안정적으로 유지됨

---

### F. UART1 송신 mutex 적용

수정:
- `proto_uart1_send_packet()`에서 `uart_tx_mutex` 획득 후 송신

효과:
- 다중 송신 경로 경쟁 시 프레임 섞임 위험 완화

---

### G. 제어 큐 확장

수정:
- `control_queue` 깊이 `16 -> 64`

효과:
- 명령 burst 시 큐 부족으로 인한 ACK fail 확률 감소

---

## 4. 수정 후 기대 로그 패턴

개선 기대:
- `FR` 증가 안정화
- `IV/OV` 증가 속도 완화
- 상태 응답이 짧은 구간에 과도하게 몰리는 현상 감소

해석 가이드:
- `DR > 0`: UART RX 큐 포화
- `O > 0`: 오버런
- `F/N/P > 0`: 물리 계층 품질 이슈
- `IV/OV`만 증가: 프레임 포맷/LEN/동기 문제 우선 점검

---

## 5. 검증 체크리스트

1) 정상 상태 요청 반복 (0x05, LEN=0)
- 10Hz~20Hz 요청 시 응답이 끊기지 않는지 확인
- 응답 폭주 대신 일정한 출력으로 유지되는지 확인

2) 모터 명령 burst (0x04)
- `w/a/s/d/auto/unauto`를 연속 전송
- ACK fail 비율이 줄었는지 확인

3) 오류 프레임 주입
- 잘못된 LEN/깨진 JSON 일부 주입
- 파서가 장시간 꼬이지 않고 정상 프레임으로 복귀하는지 확인

4) 지표 확인
- `Q/DR/O/F/N/P/IV/OV/R/RO/RF` 로그 추이 확인

---

## 6. 남아있는 리스크

- RPi 송신 구현에서 LEN(big-endian) 불일치, 프레임 사이 쓰레기 바이트 삽입,
  다중 프로세스 포트 점유가 있으면 여전히 `IV/OV`가 증가할 수 있음
- 이번 수정은 STM 측 내구성 강화이며, RPi 송신기 품질 점검은 별도 필요

---

## 7. 결론

이번 수정은 UART 통신 실패의 원인을 "하드웨어 에러"가 아닌
"프레임 경계/처리 경로/버스트 내구성" 관점에서 보강한 패치다.

핵심 변화:
- 파서 강건화
- 처리 경로 분리
- 응답 레이트 제어
- 큐 용량 확대

결과적으로 "들어왔다 안 들어왔다", "갑자기 응답 폭주" 현상 완화를 목표로 한다.
